<objective>
Add RandomForest to the GPU and `sample_weight` end-to-end. RF needs the BEST split over all
candidate thresholds, which on GPU means: a per-(feature,bin) privatized integer histogram →
inclusive prefix-scan over bins → argmax scored in the exact `criterion.rs` op order with the
exact `(feature, threshold_bits)` tie-break (GPU-04). To make the GPU-vs-CPU bit-exact gate
tractable, this plan also adds an ADDITIVE binned best-split path to the CPU oracle so both
agree on bins (the binned RF split becomes canonical — the existing raw-value `best_split` is
retained, never deleted). Finally it wires `sample_weight` through a fixed-point weighted-histogram
kernel (EST-05).

After this plan a user can call `CudaBackend.fit(x, y, cfg)` with `Algo::RandomForest`
(classifier + regressor, bootstrap, optional `sample_weight`) and get a forest byte-identical to
the CPU oracle's binned path on a fixed seed.

Purpose: GPU-04 is where Phase-3's `BinnedMatrix` finally becomes load-bearing — the scan over
bins is the RF primitive. Keeping weights as fixed-point integers is what preserves the
integer-only sibling subtraction (Plan 02) and the DET-01 determinism foundation. Scoring
cumulative counts host-side with the shipped `cpu::criterion` keeps the bit-exact gate reachable
without device-float divergence (Phase-4 locked decision, carried forward).

Output: the additive `best_split_binned` CPU path, the `RF_BINNED_HIST_SRC` / `RF_SCAN_ARGMAX_SRC`
/ `WEIGHTED_HIST_SRC` kernels, `rf_split.rs` (binned best-split launch + host scoring), the
fixed-point `sample_weight.rs`, the scheduler RF dispatch, and the `parity_rf_cpu_gpu.rs` gate.

## Assumptions (locked decisions — revisitable; each carries its alternative)

- **RF bit-exact contract = binned split is canonical (LOCKED, cross-phase, locked-decision-2):**
  the GPU RF best-split runs over Phase-3 bins; to make the bit-exact GPU-vs-CPU gate tractable we
  ADD a binned best-split path to the CPU oracle and keep the existing raw-value `best_split`.
  **This plan edits the Phase-2 CPU oracle (split_rf.rs) — flagged clearly.** The new binned path
  must be re-validated against Phase-2's distributional/KS parity gate (see verification + the study
  in Plan 06). (Alternative rejected: keep the raw-value oracle authoritative and make the GPU
  evaluate raw-value candidates — makes the GPU gate intractable at scale.)
- **sample_weight = fixed-point integer accumulation (LOCKED, locked-decision-3):** weights are
  scaled to u64 fixed-point and accumulated with integer atomicAdd; converted to f32 once at
  scoring. (Alternative rejected: float weighted atomics — non-deterministic and non-subtractable.)
  If fixed-point precision proves insufficient for `check_estimator`'s sample-weight-equivalence
  check, Plan 05 documents a 1-ULP `expected_failed_checks` entry with the reason.
- **RF scoring = host-side from cumulative counts (LOCKED, carried from Phase 4):** the scan returns
  cumulative left/right counts; scoring uses `cpu::criterion`. (Alternative: on-device `-fmad=false`
  scoring — deferred; a Phase-7 perf concern only if host round-trips dominate.)
</objective>
