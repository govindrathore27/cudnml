<objective>
Expose the whole engine through the four sklearn drop-in estimators (EST-01..06). Promote the
test-only `pyseam.rs` into a real estimator seam (fit / predict / predict_proba / fitted-attr
payloads incl. real MDI), then build the greenfield `python/sylva/` package: a `_SylvaForest`
base (no `__init__` logic, fitted attrs, `_validate_data`), the four concrete classes with correct
ET/RF x clf/reg defaults, honest `device` dispatch (no silent fallback), and the `check_estimator`
CI gate.

After this plan a user does `from sylva.ensemble import RandomForestClassifier`, fits it, and
calls `predict_proba` / `feature_importances_` — a real sklearn-compatible estimator backed by the
Rust core, and `check_estimator` is green in CI.

Purpose: EST-01..06 are the user-facing payoff of the whole phase. The Python class over a thin
Rust seam (locked-decision-4) makes `check_estimator` tractable (Python-native introspection for
get/set_params/clone) while the seam keeps the training in Rust. Honest dispatch previews the
DET-03 contract (full `execution_report_` is Phase 6). `check_estimator` runs with `device='cpu'`
so GPU-less CI can run the API gate.

Output: the promoted `pyseam.rs` estimator functions, `python/sylva/{__init__,_base,ensemble,_dispatch}.py`,
and the three pytest gates (`test_check_estimator.py`, `test_estimator_api.py`, `test_feature_importances.py`).

## Assumptions (locked decisions — revisitable; each carries its alternative)

- **Estimator surface = Python BaseEstimator subclasses over a Rust seam (LOCKED, locked-decision-4):**
  four classes in a greenfield `python/sylva/`, NO logic in `__init__`, fitted attrs
  trailing-underscore, `_validate_data` sets `n_features_in_`; wrap the Rust CpuBackend/CudaBackend
  via the pyseam pattern (`sylva_error_to_pyerr` + `py.detach()`, no `.unwrap()`, honest dispatch).
  (Alternative rejected: a Rust `#[pyclass]` estimator — reproducing get_params/clone semantics in
  PyO3 is far harder to make check_estimator-clean than native Python.)
- **Seam handle = JSON IR string (LOCKED for MVP):** fit returns the opaque JSON IR handle (the
  shipped pyseam pattern); predict/predict_proba/importances take it back. (Alternative: an in-memory
  opaque handle to avoid per-call ser/de — noted as a Phase-7 perf option; the JSON handle is correct
  and already proven.)
- **check_estimator runs device='cpu' (LOCKED):** the API gate runs GPU-less so CI without a GPU can
  enforce it; the GPU path is exercised by the Rust bit-exact gates (Plans 02-04). (Alternative: run
  on the dev GPU — slower, not CI-portable; the CPU path is the same estimator code.)
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Python user → estimator fit/predict | untrusted X/y shape/dtype/NaN, sample_weight, class_weight, max_samples, params validated via _validate_data + the seam before training |
| Python estimator → Rust seam (FFI) | the cfg dict + arrays cross PyO3; the seam re-validates and maps every error to a typed PyErr |
| device dispatch | device='cuda' unmet must RAISE — never silently degrade to CPU |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-16 | Tampering | untrusted X/y (NaN, wrong dtype, label out of range) at fit | mitigate | _validate_data + the seam's shape/label validation raise ValueError before any training (V5). |
| T-05-17 | Tampering | silent CPU fallback when device='cuda' unavailable → wrong device, hidden | mitigate | get_backend raises RuntimeError when device='cuda' + fallback='error' and CUDA is absent; only device='auto' falls back. The differentiator. |
| T-05-18 | Tampering | logic/validation in __init__ → check_estimator failure / hidden state | mitigate | __init__ stores params verbatim only; validation is in fit; the check_estimator gate is the proof. |
| T-05-19 | Tampering | untrusted sample_weight / class_weight (NaN/neg/len-mismatch) | mitigate | validated at the estimator boundary + the seam → ValueError before training. |
| T-05-20 | (Legal/IP) | copying sklearn estimator source for the Python classes | mitigate | the classes follow the public BaseEstimator developer-guide contract (Apache-2.0), reimplemented; no copied sklearn internals. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | no new packages — sklearn/numpy/scipy already in .venv-parity; the wheel uses the pinned PyO3/maturin/numpy from Phases 1-2; audit N/A. |
