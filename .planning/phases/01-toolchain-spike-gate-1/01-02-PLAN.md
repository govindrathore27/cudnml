<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Rust host -> CUDA device (FFI/launch) | Host array shape/length crosses into a kernel launch; the only place `unsafe` is used |
| CUDA-C source string -> NVRTC | Trivial original kernel source compiled at runtime (Apache-2.0; no copied restrictively-licensed source) |

Local numerical spike: no network, no auth, no user data, no persistence. Surface is memory-safety at the CUDA FFI boundary, validated by compute-sanitizer.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-03 | Tampering / DoS | unvalidated array length/bin index at the launch boundary | mitigate | Validate `n` and `bins[i] < 256` at the Rust boundary before launch; `if (i < n)` guard in the kernel (01-RESEARCH.md V5 control) |
| T-01-04 | Tampering | `unsafe` Rust at the CUDA launch FFI | mitigate | Minimize `unsafe` to the launch call; propagate cudarc Results (no `.unwrap()`); run compute-sanitizer memcheck to prove no OOB/UB |
| T-01-05 | (license) | copied restrictively-licensed kernel source | mitigate | Spike kernels are trivial original code; Apache-2.0 only (CLAUDE.md) — never copy GPL/Snap ML source |

No high-severity threats: no network listener, no untrusted external input (inputs are fixed test fixtures), no secrets. compute-sanitizer (TOOL-02) is itself the memory-safety control for this plan's FFI surface.
</threat_model>
