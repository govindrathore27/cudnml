<objective>
Lock the oracle's trustworthiness: property-based invariants over all four estimators (EST-07 property side), the documented parity contract `docs/PARITY.md` (ENG-04), and a `split_statistics` extractor the Phase-05 KS harness consumes. After Plans 02–03 the CpuBackend trains and predicts; this plan PROVES it satisfies the structural and determinism invariants that make it a bit-level oracle, and writes down exactly what parity means (and does not mean) so no one later chases bit-identity with sklearn.

Purpose: The oracle's whole value is that later GPU results can be verified against it. That requires (1) byte-level seed determinism (the contract the Phase-4 GPU path is held to — `GPU == CPU` bit-exact), (2) structural invariants that pre-validate downstream consumers (cover-count consistency pre-validates SHAP; round-trip pre-validates export), and (3) an explicit written contract distinguishing Sylva-internal bit-determinism from sklearn distributional equivalence. The `split_statistics` extractor is the seam the Plan-05 Python harness uses to run KS tests without re-deriving split data.

Output: `tests/invariants.rs` (proptest), `tests/determinism.rs` (byte-identity + order-independence), `src/parity.rs` (`split_statistics`), and `docs/PARITY.md`.
</objective>
