<objective>
Slice 2 of Phase 6 (vertical, MVP): the honest-dispatch contract end-to-end. A pure Rust decision function `resolve_backend(requested, fallback, cuda_available)` selects the backend and records the reason — and crucially RAISES `SylvaError::DeviceUnavailable` when `device="cuda"` is requested on a machine with no usable GPU under `fallback="error"`, never silently degrading to CPU. The device probe is cudarc's `CudaContext::new(0).is_ok()` (a clean `Result`, no panic, no shell-out). The Python estimator layer stores the new `device`/`fallback`/`deterministic` params verbatim (check_estimator-clean) and resolves them in `fit`, threading the decision into the `ExecutionReport` from Plan 01.

Purpose: DET-03 is a core project differentiator — Sylva refuses to silently fall back to CPU when the user asked for CUDA, unlike cuml.accel / H2O4GPU. This slice makes the refusal a typed, testable contract. It also begins filling DET-04's report (selected_backend + selection_reason + fallback_status) using the struct Plan 01 defined.

Output: `crates/sylva-core/src/dispatch.rs` (new pure decision types + fn), `availability.rs` probe in sylva-cuda, the Python `_dispatch.py` enforcement + `_base.py` param storage, and GPU-less Rust + Python tests proving the RAISE-on-unmet-cuda and auto/cpu selection. After this plan, a user requesting `device="cuda"` with no GPU gets a `RuntimeError` matching "no usable CUDA device" — verifiable on a GPU-less CI runner via a monkeypatched probe.

This plan implements orchestrator-locked Assumption D6-4 (`fallback="error"` only in MVP; `resolve_backend` is a pure typed-Result decision fn; probe via `CudaContext::new(0).is_ok()`; `device="cuda"` unmet RAISES `DeviceUnavailable`; any other `fallback` value raises `UnsupportedConfig`; NO silent CPU fallback anywhere).
</objective>

## Phase Goal

**As a** ML practitioner using Sylva from Python, **I want to** have `device="cuda"` RAISE a clear error when no usable GPU is present (instead of quietly training on CPU), **so that** I can trust that a GPU request is honored or refused, never silently downgraded — and read in `execution_report_` exactly which backend ran and why.

## Assumptions (orchestrator-resolved, no CONTEXT.md)

- **D6-4 (LOCKED):** `fallback="error"` only in MVP. `resolve_backend(requested, fallback, cuda_available)` is a pure typed-Result decision fn; probe via `CudaContext::new(0).is_ok()` (no shell-out); `device="cuda"` unmet → `SylvaError::DeviceUnavailable` RAISE; any other `fallback` value raises `UnsupportedConfig`. NO silent CPU fallback anywhere. Alternative considered: a `warn` mode that uses CPU but logs — deferred to v2 (RESEARCH OQ4); accept the param shape but RAISE on any non-"error" value.
- **D6-3 (LOCKED):** `deterministic` default = True — applied here where the Python estimator constructs its config (the `_base.py` constructor default is `deterministic=True`). The Rust field (Plan 01) is plain; the default lives at the API surface.

| Boundary | Description |
|----------|-------------|
| Python caller → `_dispatch.resolve_backend` | Untrusted `device`/`fallback` strings cross here; whitelist-validated to typed errors before any device work. |

| Chosen backend `fit` → CUDA | A CUDA error here is `?`-propagated to a typed RAISE, never a CPU retry. |
