<objective>
Slice 1 of Phase 6 (vertical, MVP): the `ExecutionReport` substrate end-to-end at the device-neutral Rust layer. Define the one report struct both backends will populate (superset of the shipped `QuantizeReport`), add the two new typed error variants the dispatch contract needs, and add the `deterministic` config field that the determinism slice toggles on. This is the foundation both later slices import: dispatch (Plan 02) records its decision *into* this report and raises `DeviceUnavailable`; the determinism slice (Plan 03) reads `cfg.deterministic` and surfaces the `deterministic` flag in the report.

Purpose: DET-04 requires `execution_report_` to record selected backend + reason, every input conversion, bytes transferred, and fallback status. DET-01/DET-03 need a place in config + error types respectively. Establishing the device-neutral types first (single source of truth, serde-tested in Rust) is what prevents the report drifting across the Python boundary later.

Output: `crates/sylva-core/src/report.rs` (new), extended `error.rs`, extended `config.rs`, re-exports in `lib.rs`, plus Rust unit tests (serde round-trip + per-field assertions). After this plan a Rust caller can construct a full `ExecutionReport`, round-trip it through `serde_json`, and pattern-match the two new `SylvaError` variants.

This plan implements orchestrator-locked Assumptions D6-5 (ExecutionReport extends QuantizeReport), D6-3 (deterministic default = True — encoded as the default this config field gets when the Python estimator constructs it; the Rust field itself is plain bool), and the error-type half of D6-4 (DeviceUnavailable variant for the fallback="error" contract).
</objective>

## Phase Goal

**As a** ML practitioner using Sylva from Python, **I want to** read a single `execution_report_` after every fit that explains which backend ran, why, every input conversion, and the bytes transferred, **so that** I can audit and trust the device decision and never be surprised by a silent dtype/layout copy or backend substitution.

(ROADMAP `**Goal:**` for Phase 6 is "Layer bit-reproducible determinism onto the now-correct forest and enforce the no-silent-fallback dispatch contract." Reframed as the practitioner-facing user story above per MVP mode; this slice delivers the report substrate that makes the decision auditable.)

## Assumptions (orchestrator-resolved, no CONTEXT.md)

- **D6-5 (LOCKED):** `ExecutionReport` extends `QuantizeReport` (device-neutral serde struct, nesting the existing report). Alternative considered: Python-side dict — rejected because it can't be unit-tested in Rust and drifts from backend truth.
- **D6-3 (LOCKED):** `deterministic` default = True (safe mode is the default). Alternative: default False for speed — rejected; the project's correctness stance is bit-reproducibility-by-default. (The Rust `TrainConfig.deterministic` field is a plain bool; the *default of True* is applied where the Python estimator constructs the config — Plan 02/03 — but the field lands here.)
- **D6-4 (LOCKED, error-type half):** unmet `device="cuda"` → new `SylvaError::DeviceUnavailable` RAISE; any other `fallback` value → `UnsupportedConfig`. Alternative: reuse `InvalidConfig` — rejected; a distinct variant lets the Python layer map "you asked for cuda but there is none" to a precise message and lets tests match on it.
